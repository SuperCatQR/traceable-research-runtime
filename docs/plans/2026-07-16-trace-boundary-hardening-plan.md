# Trace 边界强化计划

- 状态：提议中，尚未执行
- 日期：2026-07-16
- 前置状态：[三级 Trace 展示计划](../history/plans/2026-07-16-three-level-trace-presentation-plan.md) 已实施

## 已确认的产品边界

用户只通过自然语言与模型对话。模型内部生成并冻结 `ResearchBrief`，但它不是用户表单，也不需要
用户确认。Trace 用于解释系统已经做过什么，而不是要求用户参与流程或展示隐藏推理。

| 级别 | 面向谁 | 必须提供 | 必须排除 |
| --- | --- | --- | --- |
| L1 聊天 | 所有用户 | 自然对话、紧凑运行状态、答案、必要来源 | Brief、模型 ID、轮次技术元数据、查询、理由、原始事件 |
| L2 研究概览 | 想了解过程的对话所有者 | 问题理解、检索覆盖、主要来源、综合/失败摘要 | 原始事件集、全部搜索结果、完整快照、隐藏推理 |
| L3 审计详情 | 需要复盘或诊断的对话所有者 | 分页、可筛选、带时间和稳定顺序的审阅安全事件 | 系统提示词、密钥、模型原始输入、完整网页正文、隐藏推理 |

当前 v6 已实现 L1/L2/L3 的展示分层、拥有权校验、L2 白名单投影、L3 分页筛选和右侧抽屉；本计划
处理其余的数据最小化与审计完整性问题。

## 为什么需要下一阶段

当前聊天界面不渲染内部内容，但 Conversation 的 HTTP 响应仍携带部分仅应在 L2/L3 使用的研究数据。
这使“界面简洁”停留在前端约定，不能成为接口保证。另一方面，现有 L3 是审阅安全的事件摘要，缺少
每条事件的稳定序号、发生时间、搜索引擎选择/回退，以及研究停止原因，难以回答“何时、为何发生”。

## 设计决策

新 Trace v7 采用一个深模块接口：核心只写完整、顺序确定的审计事件；Host 在验证拥有权后产生两种
不可逆投影；浏览器只能访问聊天投影、概览投影和审计投影。浏览器永远不接触原始 JSONL，也不承担
脱敏责任。

不做 v6 到 v7 的线上兼容或迁移。v6 数据保留为历史审计材料；v7 使用新的运行目录或持久卷，不能
混写。这样避免一个 schema 同时承载两套事件语义，但不授权删除旧数据。

## 实施范围

1. **最小聊天投影**
   - 在 `src/runtime.rs`、`demo-host/src/workspace_api.rs` 和
     `demo/src/research-workspace-client.ts` 拆出 Chat Turn/Answer DTO。
   - 聊天端点只返回 L1 所需字段；knowledge draft、主张理由、比较数据、快照引用等只能通过 L2/L3
     投影取得。
   - 保留服务端完整答案以支持重放和审计，不把完整内部对象当作公共聊天 DTO。

2. **可排序、可定时的审计事件**
   - 在 `src/research_trace.rs` 定义 v7 事件 envelope：`sequence`、`occurred_at`、事件种类和事件内容。
   - 为准备、模型调用、查询、搜索、归档、选源、综合和失败事件写入发生时间。
   - L3 返回 sequence/timestamp，并按稳定顺序分页；L2 继续只返回压缩概览。

3. **搜索决策与停止原因**
   - 在 `src/research_domain.rs`、`src/external_adapters.rs`、`src/research_run.rs` 记录搜索引擎、
     每次尝试、失败类别和回退理由；此项与
     [Google 优先、Bing 回退待办](../todo/2026-07-16-google-first-searxng-search-fallback.md) 一起实施。
   - 新增可审计的探索停止事件，记录达到轮数/输入预算/快照上限、无新 URL 或正常完成等原因。

4. **审阅安全投影与右侧栏**
   - 扩展 `demo-host/src/workspace_api.rs` 的 L2/L3 投影，不扩大原始事件暴露范围。
   - 在 `demo/src/main.ts` 与 `demo/src/styles.css` 中让右侧栏显示事件时间、搜索引擎/回退和停止原因；
     L1 保持无技术元数据的普通对话。
   - 继续保留懒加载、所有权 404、阶段筛选、分页和窄屏抽屉。

5. **验证与部署**
   - 为 v7 回放、序号连续性、时间排序、字段白名单、L1 不含内部字段、搜索回退和停止原因补充核心、
     Host 与 HTTP 验证。
   - 更新 ADR、架构文档、运行环境变量和部署脚本，默认切换到新的 v7 运行目录/卷；旧卷不删除。

## 执行顺序

1. 先固定 v7 事件与 Chat/L2/L3 DTO 契约，并写拒绝泄露内部字段的测试。
2. 再让运行时和搜索适配器产生新事件、回放事件和停止事件。
3. 然后实现 Host 投影与右侧栏展示，逐项对照三层表格。
4. 最后使用新的空数据目录进行完整 HTTP 验证、WSL/服务器部署和人工界面验收。

## 验收标准

- 抓取聊天响应不能得到 Brief、query、knowledge draft、claim rationale、raw event 或 API key。
- 同一条 L3 事件在重放后拥有相同 sequence，且时间顺序可解释。
- 每一次 Google/Bing 搜索选择、失败和回退可从 L3 复盘，但不会泄露模型输入。
- 每次研究停止均有一个可阅读、可审阅的原因。
- 右侧栏关闭时不请求 L2/L3；打开概览只请求 L2；切换审计才请求 L3。
- v7 不读取或追加 v6 数据，部署脚本不删除旧卷。
